# FormStream 部署文档

> 本文档记录 FormStream 从零开始部署到 Cloudflare Workers + Supabase 的完整流程，也可以作为**重建/迁移/排障**时的参照手册。按顺序做完即可上线；每一步都标注了"做完之后应该看到什么"，方便核对。

| 项目 | 内容 |
|---|---|
| 部署形态 | 单个 Cloudflare Worker（React Router v7 SSR + Hono API 同体） |
| 数据库/鉴权 | Supabase（Postgres + Auth） |
| 当前生产 Supabase 项目 | `https://fccxmikziiimwwgxfdwf.supabase.co` |
| 当前生产 Cloudflare 资源 | R2 桶 `formstream-uploads`、KV `RATE_LIMIT`、Queue `formstream-notify` |

## 1. 前置条件

- Node.js（建议 20+）、npm
- 一个 Cloudflare 账号，已绑定支付方式（R2/KV 首次启用通常需要，即使用量在免费额度内）
- 一个 Supabase 账号
- 本机能执行 `npx wrangler` 命令（项目 `devDependencies` 里已经带了 wrangler，不需要全局安装）

克隆/进入项目目录后先装依赖：

```bash
cd formstream
npm install
```

## 2. Supabase 端设置

### 2.1 建项目

1. 打开 supabase.com → New Project
2. 项目名：`formstream`（如果要区分环境，建 `formstream-dev` / `formstream-prod` 两个项目）
3. 数据库密码：随机生成一个强密码，存进密码管理工具——**这个项目本身用不到这个密码**（连库走 `supabase-js`/REST API，不走裸 Postgres TCP），只在你需要直连数据库做运维时才用得上
4. Region：选离目标用户最近的区域，建好之后不好迁移
5. Plan：开发阶段 Free 没问题；**正式上线前必须升级到 Pro**——Free 档闲置 7 天会自动暂停项目，不适合生产环境

### 2.2 初始化数据库（SQL Editor 里执行）

打开 Supabase 控制台 → SQL Editor，按顺序执行下面三段 SQL。

**第一段：建表**

```sql
create table public.profiles (
  id          uuid primary key references auth.users(id) on delete cascade,
  role        text not null default 'user'   check (role in ('user','admin')),
  status      text not null default 'active'  check (status in ('active','suspended')),
  plan        text not null default 'free',   -- free / pro / business
  created_at  timestamptz not null default now()
);

create table public.forms (
  id              uuid primary key default gen_random_uuid(),
  user_id         uuid not null references auth.users(id) on delete cascade,
  name            text not null,
  access_key      text not null unique default encode(gen_random_bytes(16),'hex'),
  allowed_domains text[] not null default '{}',
  is_active       boolean not null default true,
  turnstile_enabled boolean not null default false,
  redirect_url    text,
  spam_protection boolean not null default true,
  created_at      timestamptz not null default now(),
  updated_at      timestamptz not null default now()
);
create index idx_forms_user on public.forms(user_id);

create table public.notification_channels (
  id          uuid primary key default gen_random_uuid(),
  form_id     uuid not null references public.forms(id) on delete cascade,
  type        text not null check (type in ('dingtalk','feishu','wework','email','webhook')),
  config      jsonb not null,        -- {webhook_url, secret, emails:[], ...}
  is_active   boolean not null default true,
  created_at  timestamptz not null default now()
);
create index idx_channels_form on public.notification_channels(form_id);

create table public.submissions (
  id          uuid primary key default gen_random_uuid(),
  form_id     uuid not null references public.forms(id) on delete cascade,
  data        jsonb not null,
  files       jsonb not null default '[]',
  ip          inet,
  user_agent  text,
  country     text,
  is_spam     boolean not null default false,
  created_at  timestamptz not null default now()
);
create index idx_sub_form_created on public.submissions(form_id, created_at desc);

create table public.usage (
  user_id     uuid not null references auth.users(id) on delete cascade,
  period      date not null,
  count       integer not null default 0,
  primary key (user_id, period)
);

create table public.audit_logs (
  id          uuid primary key default gen_random_uuid(),
  actor_id    uuid references auth.users(id) on delete set null,
  actor_email text,
  action      text not null,
  target_type text not null,
  target_id   text,
  detail      jsonb not null default '{}',
  created_at  timestamptz not null default now()
);
create index idx_audit_logs_created on public.audit_logs(created_at desc);

-- 注册后自动建 profile
create function public.handle_new_user() returns trigger language plpgsql security definer as $$
begin
  insert into public.profiles (id) values (new.id);
  return new;
end; $$;
create trigger on_auth_user_created
  after insert on auth.users for each row execute function public.handle_new_user();
```

**第二段：开 RLS（行级安全）**

```sql
alter table public.profiles enable row level security;
alter table public.forms enable row level security;
alter table public.submissions enable row level security;
alter table public.notification_channels enable row level security;
alter table public.usage enable row level security;
alter table public.audit_logs enable row level security;

create policy "own profile" on public.profiles
  for select using (auth.uid() = id);

create policy "own forms" on public.forms
  for all using (auth.uid() = user_id);

create policy "own submissions" on public.submissions
  for all using (exists (
    select 1 from public.forms f
    where f.id = submissions.form_id and f.user_id = auth.uid()));

create policy "own channels" on public.notification_channels
  for all using (exists (
    select 1 from public.forms f
    where f.id = notification_channels.form_id and f.user_id = auth.uid()));

create policy "own usage" on public.usage
  for select using (auth.uid() = user_id);

-- audit_logs 不建任何 policy：RLS 默认拒绝所有人，只有 service_role（绕过 RLS）能读写
```

> ⚠️ 跑完之后务必去 **Table Editor** 里确认这 6 张表真的出现了。Supabase 的 PostgREST schema cache 偶尔会和实际建表状态不同步，跑完 SQL 没报错也不代表表已经能被 API 访问——如果之后调用接口报 `Could not find the table 'xxx' in the schema cache`（错误码 `PGRST205`），先去 Table Editor 肉眼确认表是否存在，再决定要不要重新跑。

**第三段：（可选）确认触发器生效**

```sql
select tgname from pg_trigger where tgname = 'on_auth_user_created';
```
能查到一行说明触发器建好了——之后每个新注册用户都会自动在 `profiles` 表里生成一条记录。

### 2.3 Auth 配置

1. Authentication → Providers：确认 **Email** 登录方式已启用（默认就是开的）
2. Authentication → URL Configuration：
   - Site URL：填你的正式域名（或者部署后拿到的 `*.workers.dev` 子域名）
   - Redirect URLs：和 Site URL 保持一致即可，这个项目目前没有用到第三方 OAuth 回调
3. 邮件模板：用 Supabase 自带的默认模板即可（确认邮件、找回密码邮件），暂不需要自定义 SMTP
4. 注意 Supabase 自带邮件发送有频率限制（免费档每小时几封），开发阶段频繁注册测试账号容易触发 "email rate limit exceeded"——这是正常现象，等一会儿或者用 Admin API 直接创建已确认的测试账号即可（见 §11 故障排查）

### 2.4 拿到三个关键值

Settings → API：

| 名称 | 在哪 | 用途 |
|---|---|---|
| Project URL | Settings → API → Project URL | 写进 `wrangler.json` 的 `vars.SUPABASE_URL`（非机密） |
| Publishable key（旧称 anon key） | Settings → API → API Keys，`sb_publishable_...` | `SUPABASE_ANON_KEY`，验用户 JWT / SSR session 用，非机密 |
| Secret key（旧称 service_role key） | 同上，`sb_secret_...` | `SUPABASE_SERVICE_KEY`，**绕过 RLS 的最高权限密钥，绝密** |

> Supabase 新版控制台把 `anon key` 改名叫 **Publishable key**，`service_role key` 改名叫 **Secret keys**（且支持创建多个，建议给这个项目单独建一个，方便以后单独吊销轮换）。这两个名字不一样但本质就是原来的 anon/service_role。

## 3. Cloudflare 端设置

### 3.1 账号与计费

- R2 / KV 首次使用通常要求账号已绑定支付方式（即使用量在免费额度内）
- **Cloudflare Queues 现在 Free 计划也能直接用**（不需要升级 Paid）：Free 档每天 10,000 次 operations 免费、消息保留期固定 24 小时；Paid 计划（$5/月起）每月 100 万次免费、保留期可配到 14 天。开发/早期阶段用 Free 就够，等通知量真的上来了再考虑升级

### 3.2 创建 R2 bucket

```bash
npx wrangler r2 bucket create formstream-uploads
```
（生产环境已经叫 `formstream-uploads`，如果是全新部署可以换别的名字，但要和下面 `wrangler.json` 里的 `bucket_name` 保持一致）

### 3.3 创建 KV namespace

```bash
npx wrangler kv namespace create RATE_LIMIT
```
命令会返回一个 `id`，记下来，下一步要填进 `wrangler.json`。

### 3.4 创建 Queue

```bash
npx wrangler queues create formstream-notify
```

### 3.5 （可选）Turnstile 人机验证

1. Cloudflare Dashboard → Turnstile → Add site
2. Domain 填你的正式域名（没有就先填占位域名，部署后再回来改）
3. 拿到 **Site Key**（前端表单页要嵌入用，非机密）和 **Secret Key**（后端校验用，配成 secret `TURNSTILE_SECRET`）
4. 这一步可以跳过，但在配置 `TURNSTILE_SECRET` 前不能启用 `form.turnstile_enabled`；如果已有记录处于启用状态但密钥缺失，公开提交接口会返回 503，绝不会绕过验证

### 3.6 自定义域名（可选，建议生产环境配）

1. 域名要先加到 Cloudflare（NS 切过来）
2. Workers & Pages → 找到 `formstream` 这个 Worker → Settings → Domains & Routes → 绑定自定义域名
3. 没有域名也不影响部署，先用 Cloudflare 分配的 `formstream.<你的账号>.workers.dev` 子域名

### 3.7 Resend（邮件通知渠道，可选）

1. 注册 resend.com
2. Domains → 添加发信域名，按提示在 DNS 里加 SPF/DKIM 记录（如果域名托管在 Cloudflare，直接在 Cloudflare DNS 里加）
3. 拿到 `RESEND_API_KEY`
4. 没有自己验证的域名也能先用 `onboarding@resend.dev` 当发件地址跑通流程（代码里 `workers/lib/notify-email.ts` 默认就是这个），正式上线前换成自己验证过的域名

## 4. 项目里的 wrangler 绑定配置

把上一步拿到的 R2/KV/Queue 名字和 id，以及 Supabase Project URL，填进项目根目录的 `wrangler.json`：

```jsonc
{
  "name": "formstream",
  "main": "./workers/app.ts",
  "compatibility_date": "2025-10-08",
  "compatibility_flags": ["nodejs_compat"],
  "observability": { "enabled": true },
  "upload_source_maps": true,
  "vars": {
    "SUPABASE_URL": "https://你的项目id.supabase.co"
  },
  "r2_buckets": [
    { "bucket_name": "formstream-uploads", "binding": "UPLOADS" }
  ],
  "kv_namespaces": [
    { "binding": "RATE_LIMIT", "id": "你的 KV namespace id" }
  ],
  "queues": {
    "producers": [{ "binding": "NOTIFY_QUEUE", "queue": "formstream-notify" }],
    "consumers": [{ "queue": "formstream-notify", "max_batch_size": 10, "max_retries": 5 }]
  }
}
```

改完之后重新生成类型（`worker-configuration.d.ts` 是自动生成的，绑定改了就要重跑）：

```bash
npm run cf-typegen
```

这个命令做两件事：`wrangler types` 重新生成 Cloudflare 绑定的类型，`react-router typegen` 重新生成路由类型。**改完 `wrangler.json` 或新增/修改路由之后都要重跑这个命令。**

## 5. 配置密钥（Secrets）

项目需要 5 个密钥，分两套环境配置方式：

| 名称 | 说明 | 是否机密 |
|---|---|---|
| `SUPABASE_ANON_KEY` | Supabase Publishable key | 否（可下发前端，但仍按密钥管理） |
| `SUPABASE_SERVICE_KEY` | Supabase Secret key，绕过 RLS | **是，绝密** |
| `RESEND_API_KEY` | 发邮件通知用 | 是 |
| `ADMIN_EMAILS` | 逗号分隔的管理员邮箱，登录时自动提权 | 否，但别公开 |
| `TURNSTILE_SECRET` | Turnstile 校验密钥（可选） | 是 |

### 5.1 本地开发：`.dev.vars`

在项目根目录创建 `.dev.vars`（已经被 `.gitignore` 的 `.dev.vars*` 规则排除，不会进 git）：

```
SUPABASE_ANON_KEY=sb_publishable_xxxxx
SUPABASE_SERVICE_KEY=sb_secret_xxxxx
ADMIN_EMAILS=you@example.com
RESEND_API_KEY=re_xxxxx
```

`TURNSTILE_SECRET` 没配的话留空不写就行，代码里是按可选处理的。

`npm run dev` 会自动读取这个文件。

### 5.2 生产环境：`wrangler secret put`

部署到 Cloudflare 之后，生产环境的密钥**不**放在 `.dev.vars`（那个文件只有本地 `wrangler dev`/`react-router dev` 会读），要逐个用命令写进 Cloudflare：

```bash
npx wrangler secret put SUPABASE_ANON_KEY
npx wrangler secret put SUPABASE_SERVICE_KEY
npx wrangler secret put ADMIN_EMAILS
npx wrangler secret put RESEND_API_KEY
npx wrangler secret put TURNSTILE_SECRET   # 可选，没有就不用跑这条
```

每条命令会交互式提示你输入值（不会回显在终端历史里），输入完按回车即可。这些密钥只存在 Cloudflare 那边，不会出现在任何代码文件或 git 历史里。

> 想确认已经配置了哪些 secret（看不到值，只能看名字）：`npx wrangler secret list`

## 6. 本地开发与验证

```bash
npm run dev
```

打开 `http://localhost:5173`，走一遍核心流程：

1. 注册一个账号（用 `ADMIN_EMAILS` 里配的邮箱，确认提权逻辑）
2. 因为 Supabase 默认开邮箱确认，注册后可能要去邮箱点确认链接才能登录（或者参考 §11 用 Admin API 直接创建已确认账号）
3. 登录后建一个表单，拿到 `access_key`
4. 用 curl 或者真实页面提交一条测试数据：
   ```bash
   curl -X POST http://localhost:5173/s/<access_key> -d "name=test" -H "Accept: application/json"
   ```
5. 后台能看到这条提交记录，说明入口分流、数据库写入、RLS 都正常

本地 `wrangler dev`/`react-router dev` 也会模拟 Queue 消费，终端日志里能看到 `QUEUE formstream-notify 1/1 (...)`，说明通知队列也在本地跑通了。

## 7. 构建与正式部署

### 7.1 先跑一次完整检查

```bash
npm run check
```
这条命令依次做：`tsc` 类型检查 → `react-router build` 构建 → `wrangler deploy --dry-run` 模拟部署（不会真的发布，只验证配置/绑定是否正确）。三步都过了才说明能放心部署。

### 7.2 正式部署

```bash
npm run deploy
```
等价于 `wrangler deploy`：构建并把 Worker 发布到 Cloudflare。命令执行完会打印出访问地址（`https://formstream.<你的账号>.workers.dev` 或绑定的自定义域名）。

### 7.3 部署成功的标志

终端输出类似：
```
Uploaded formstream (x.xx sec)
Deployed formstream triggers (x.xx sec)
  https://formstream.<account>.workers.dev
```

## 9. 首个管理员账号

不需要在 Supabase 控制台手动改 `role` 字段。流程是：

1. 把你要当管理员的邮箱写进 `ADMIN_EMAILS` secret（逗号分隔多个）
2. 用这个邮箱正常注册/登录一次
3. 登录成功的那一刻（`app/routes/login.tsx` 的 action 里），代码会自动检查邮箱是否在 `ADMIN_EMAILS` 列表里，如果是且当前还不是 admin，就把 `profiles.role` 改成 `admin`
4. 之后这个账号登录就能看到"管理面板"入口

后续要增减管理员，去 `/admin/users` 页面里直接改对应用户的 `role`，或者直接在 Supabase 控制台改 `profiles.role` 字段。

## 10. 日常更新发布流程

代码改完之后，正常发布一次更新：

```bash
npm run cf-typegen   # 如果改了 wrangler.json 绑定或新增了路由
npm run check        # 类型检查 + build + dry-run，确认没问题
npm run deploy        # 正式发布
```

如果这次更新涉及新的数据库字段/表，要先去 Supabase SQL Editor 把 DDL 跑了，**再**部署代码（避免代码上线后找不到新字段/表）。

如果这次更新涉及新的 secret（比如换了 Resend key），用 `wrangler secret put <NAME>` 更新，不需要重新部署代码（secret 更新立即生效于下一次请求）。

## 11. 故障排查

| 现象 | 原因 | 处理 |
|---|---|---|
| `Could not find the table 'xxx' in the schema cache`（PGRST205） | PostgREST 没识别到刚建的表，可能是 SQL 实际没跑成功 | 去 Supabase Table Editor 肉眼确认表是否存在；不存在就重新跑 DDL，存在就等一会儿重试（schema cache 偶尔同步延迟） |
| 注册时报 `email rate limit exceeded` | Supabase 自带邮件发送频率限制，开发阶段连续注册测试账号容易触发 | 等一会儿重试；或者用下面的脚本直接通过 Admin API 建一个已确认的测试账号，跳过发信环节： |

```js
// 临时脚本：直接创建一个已确认邮箱的测试账号，不走邮件确认流程
import { createClient } from "@supabase/supabase-js";
const admin = createClient("https://你的项目.supabase.co", "你的 SUPABASE_SERVICE_KEY");
await admin.auth.admin.createUser({
  email: "test@example.com",
  password: "Test12345678",
  email_confirm: true,
});
```

| 现象 | 原因 | 处理 |
|---|---|---|
| 提交表单返回 404 `form not found or inactive` | `access_key` 错误，或表单被停用，或 KV 缓存了 60 秒前的旧状态 | 确认 access_key 没抄错；改了表单状态后最多等 60 秒缓存过期（正常改动会主动 `invalidateFormCache`，理论上不用等） |
| 提交返回 429 | 触发了限流（单 IP 20次/分，单表单120次/分） | 正常防滥用行为，等 1 分钟再试 |
| 通知没收到 | 渠道配置错了，或者是不可重试的永久失败被静默跳过 | 去 `/admin/logs` 查有没有 `通知渠道投递失败` 记录；用渠道详情页的"测试"按钮单独测一下这个渠道 |
| `wrangler deploy` 报绑定相关错误 | `wrangler.json` 里的 R2/KV/Queue 名字或 id 跟 Cloudflare 后台实际资源不一致 | 用 `npx wrangler r2 bucket list` / `npx wrangler kv namespace list` / `npx wrangler queues list` 核对名字和 id 是否一致 |
| 本地 `npm run dev` 起不来或读不到密钥 | `.dev.vars` 不存在或者键名拼错 | 对照 §5.1 检查文件内容，键名必须跟 `workers/env.d.ts` 里声明的一致 |

## 12. 回滚

Cloudflare Workers 每次 `wrangler deploy` 都会生成一个新版本。出问题需要紧急回滚：

```bash
npx wrangler deployments list   # 看历史版本
npx wrangler rollback [版本ID]  # 回滚到指定版本（不传版本ID则回滚到上一个）
```

数据库层面没有自动回滚机制——如果某次更新包含了破坏性的 DDL（比如删字段），回滚代码并不会把数据库结构改回去，这种情况要手动写反向 SQL。所以**有破坏性的 DDL 变更，上线前一定要想清楚怎么回退**，必要时先在 Supabase 加字段、跑一段时间观察没问题后再删旧字段，而不是一步到位替换。
